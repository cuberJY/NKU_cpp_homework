## 一. 作业题目

**To The Moon 拼图游戏** —— 基于 GF(2) 域高斯消元的最优解求解拼图游戏。

游戏灵感来源于 *《To The Moon》* 中的拼图小游戏：每一块拼图碎片都有**正面（图案朝上）**和**反面（空白朝上）**两种状态。玩家每次可以翻转一整行、一整列或一条对角线上的所有碎片，目标是**将所有碎片翻转到正面**。其操作机制与经典的 *Lights Out*（熄灯游戏）类似，每次操作会同时改变特定条件下格子的状态，可以转换为求解 GF(2) 域上的异或方程组。

本项目包含两种游戏模式：

- **关卡模式**：共 15 个预置关卡，难度逐步递增，完成当前关卡后解锁下一关
- **随机模式**：系统随机生成拼图，包含简单（4×4）、普通（5×4）、困难（6×5）、极限（6×6）四种难度

### 2. 算法或公式

#### 2.1 数学模型

拼图操作的本质是在 GF(2) 域（模 2 算术）上对格子状态进行异或（XOR）翻转：每行、每列、每条对角线的翻转都会使该线上所有格子取反。因此，**每个格子的最终状态 = 初始状态 XOR 影响该格子的所有操作的异或和**。目标是将全部格子翻为正面（即 1），问题等价于：已知每个格子的初始值 $b_{ij} \in \{0,1\}$，求哪种操作组合能使所有格子变为 1。

设拼图 $m$ 行 $n$ 列，全体操作为 $k = m + n + (\text{主对角线}) + (\text{副对角线})$ 种。定义布尔变量 $x_0, \dots, x_{k-1} \in \{0,1\}$，表示对应操作是否执行。对于格子 $(i,j)$，记 $S_{ij}$ 为影响该格子的操作下标集合（始终包含第 $i$ 行和第 $j$ 列，若该格子位于某对角线上则还包含该对角线），则该格子需满足：

$$\bigoplus_{p \in S_{ij}} x_p = \neg b_{ij}$$

即将 $S_{ij}$ 中对应 $x_p$ 的异或和应等于该格子"未被翻正"的补值。联立全部 $m \times n$ 个格子的方程，得到 GF(2) 域上的线性方程组 $A \mathbf{x} = \mathbf{b}$，其中 $A$ 为 $mn \times k$ 的系数矩阵，$\mathbf{b}$ 为 $mn$ 维常数列向量。

#### 2.2 高斯-约旦消元（GF(2) 域）

> **为何不用 BFS？** BFS 需要在状态空间中对每种操作分支进行广度优先搜索，状态总数为 $2^{m \times n}$。以 6×6 拼图为例，状态空间高达 $2^{36} \approx 6.87 \times 10^{10}$，无论是时间还是内存都无法承受。而 GF(2) 域高斯消元法的时间复杂度为 $O((mn)^3)$，可在毫秒级完成求解，且能保证找到全局最优解。

对于 GF(2) 域上的线性方程组，采用**高斯-约旦消元法**（Gauss-Jordan Elimination）求解。与传统实数域消元相比，GF(2) 域的特点如下：

| 实数域 | GF(2) 域 |
|--------|----------|
| 加法 `+` | 异或 `^`（即 XOR） |
| 减法 `-` | 异或 `^`（减法和加法等价） |
| 乘法 `*` | 与 `&` |
| 除法（除以标量） | 无需（元素只有 0 和 1） |

消元算法的核心步骤如下：

```
输入：增广矩阵 mat[mRow][mCol]，其中 mRow = row*col, mCol = opCount+1
输出：行简化阶梯形矩阵（Reduced Row Echelon Form）

1. 设当前处理行 pRow = 0，当前处理列 pCol = 0
2. while pRow < mRow 且 pCol < mCol:
   a. 在第 pCol 列中，从第 pRow 行开始向下寻找值为 1 的元素作为主元（pivot）
   b. 若找不到主元，则 pCol++（该列为自由变量列），continue
   c. 将主元行交换到第 pRow 行
   d. 用主元行消去该列中所有其他行：
      - 对于上方行（i < pRow）：mat[i][j] ^= mat[pRow][j]
      - 对于下方行（i > pRow）：mat[i][j] ^= mat[pRow][j]
   e. pRow++, pCol++
```

消元完成后，可区分**主元变量**（对应主元列，有唯一解）和**自由变量**（对应非主元列，取 0 或 1 均满足方程）。

#### 2.3 自由变量枚举 —— 求最少步数解

当存在自由变量时，方程组有无穷多解（$2^{\text{自由变量数}}$ 种）。为求得**最少步数**的最优解，使用**位掩码枚举**对所有自由变量的赋值组合进行遍历：

```
输入：自由变量列下标列表 Free，大小为 freeCount
输出：最少操作步数 minStep 及最优解 bestSol

1. 初始化 minStep = opCount + 1
2. for mask in [0, 2^freeCount - 1]:      // 位掩码枚举
   a. 设 step = 0
   b. 根据 mask 的二进制位给自由变量赋值，统计已赋值 1 的数量计入 step
   c. 若 step >= minStep，剪枝跳过（不可能更优）
   d. 回代求解所有主元变量，统计其中为 1 的数量累加入 step
   e. 若 step < minStep，更新 minStep 和 bestSol
3. 返回 minStep 和 bestSol
```

位掩码的效率很高：每个 mask 值对应 $[0, 2^{freeCount})$ 区间的一个整数，其二进制表示直接给出所有自由变量的赋值，无需递归。

#### 2.4 拼图随机打乱

随机模式通过**模拟玩家操作**来打乱拼图，而非直接随机生成格子状态：

```cpp
void PuzzleBoard::randomBoard(){
    uniform_int_distribution<> dis(1, row+col+20);
    for (int i=0; i<30; i++){
        int n = dis(gen);
        if (n>0 && n<=row)               turnRow(n-1);      // 行: row 个区间
        if (n>row && n<=row+col)         turnCol(n-row-1);  // 列: col 个区间
        if (n>row+col && n<=row+col+10)  if(isMD) turnMD(); // 主对角线: 10 个区间
        if (n>row+col+10)                if(isSD) turnSD(); // 副对角线: 10 个区间
    }
}
```

**模拟玩家操作** 如果直接随机填充格子的 0/1 状态，生成的拼图可能无解（即无法通过允许的操作将全部格子翻为正面）。而通过调用 `turnRow()`、`turnCol()` 等方法打乱，本质是在初始全正面状态上依次施加合法操作。由于每种操作都是可逆的（GF(2) 域上操作两次等于不操作），打乱后的状态一定可以通过反向施加这些操作还原到全正面——从而保证拼图**必然有解**。

**对角线权重**：随机数区间 `[1, row+col+20]` 中，行占 `row` 个区间、列占 `col` 个区间，而每条对角线固定占 10 个区间。以 6×6 拼图为例，行+列共 12 个区间，两条对角线各占 10 个区间，对角线被选中的概率（各约 31%）高于单行或单列（各约 19%），使得打乱后的拼图更充分地利用对角线操作，增加解法的多样性和挑战性。

打乱完成后计算 `initialMinStep`，若最优解步数少于 3 步则递归重新打乱，确保拼图具有一定挑战性。

#### 2.5 倒计时与步数限制机制

为增加游戏的挑战性和策略深度，项目在多种游戏模式中引入了**倒计时**和**步数限制**两种约束机制。

**（1）难度配置**

`BoardConfig` 结构体中的 `limitTime`（时间限制，秒）和 `limitStep`（步数限制）字段控制约束条件，`-1` 表示不启用。各模式/难度的约束配置如下：

| 模式 | 拼图大小 | 对角线 | 步数限制 | 时间限制 |
|------|----------|--------|----------|----------|
| 简单 | 4×4 | 主 | 无 | 30s |
| 普通 | 5×4 | 主 | 无 | 25s |
| 困难 | 6×5 | 主+副 | 最优+2步 | 20s |
| 极限 | 6×6 | 主+副 | 必须最优 | 20s |
| 关卡1-5 | 4×3 / 4×4 | 主 | 无 | 无 |
| 关卡6-10 | 5×4 / 6×5 / 6×6 | 主+副 | 逐步收紧 | 无 |
| 关卡11-15 | 6×6 / 6×5 | 主+副 | 0~4步 | 10~25s |

**（2）倒计时实现**

倒计时在 `BoardView` 层通过 `QTimer` 实现，每秒触发一次 `onCountdownTick()`：

```cpp
// 倒计时每秒触发
void BoardView::onCountdownTick(){
    if (mainWindow->getIsAnimation()) remainingTime--;
    else remainingTime -= 2;
    setTimeLabel();
}
```

- 视觉上始终每次刷新减 1 秒，不受动画开关影响。内部实现：开启动画时 `remainingTime` 减 1 并直接显示；关闭动画时 `remainingTime` 减 2 并以 `remainingTime / 2` 显示（因关闭动画后定时器仍按相同间隔触发，需补偿无动画延时的空白区间，保持视觉上每秒减 1）
- `setTimeLabel()` 以 `MM:SS` 格式显示剩余时间，当剩余时间 ≤ 10 秒时文字变为橙色警告
- 当 `remainingTime ≤ 0` 时，停止定时器，调用 `controller->setTimedOut(true)` 并触发 `checkGameState()`，判定为超时失败

**（3）步数限制实现**

步数限制在 `PuzzleBoard` 层维护。初始化拼图时，`limitStep` 会被调整为相对于最优解的值：

```cpp
void PuzzleBoard::setBoard(const vector<vector<bool>>& newBoard){
    board = newBoard;
    playerStep = 0;
    initialMinStep = getMinStep();
    if (limitStep >= 0) limitStep += initialMinStep;
}
```

即：若配置中限制比最优解多 N 步，则 `limitStep = initialMinStep + N`。`isFailed()` 判定如下：

```cpp
bool PuzzleBoard::isFailed() const{
    if (limitStep < initialMinStep) return false;
    if (playerStep >= limitStep) return true;
    return false;
}
```

- 若未启用步数限制（`limitStep` 为 -1），`limitStep < initialMinStep` 永远为 true，不会失败
- 每次玩家操作后 `BoardController::checkGameState()` 调用 `isFailed()`，若超步数则判定失败

**（4）UI 反馈**

`BoardView` 在界面上实时展示两类约束的状态：

- **步数标签** `remstepLabel`：显示 "剩余 X 步"，由 `updateRemStepLabel()` 通过 `controller->getRemStep()`（即 `limitStep - playerStep`）计算
- **时间标签** `timeLabel`：以 `MM:SS` 格式显示倒计时，仅在启用时间限制时可见
- **失败提示**：`checkGameState()` 根据 `isTimedOut()` 分别显示"时间到\n游戏失败！"或"步数超过限制\n游戏失败！"

#### 2.6 动画实现

为提升游戏的视觉体验和操作反馈，项目基于 Qt 动画框架实现了一套完整的动画系统，包含**翻转动画**和**完成动画**两大类。所有动画通过 `Tool` 工具类的静态方法统一创建和管理。

**（1）动画框架与工具类**

`Tool` 类封装了三个核心动画工厂方法（见 [Tool.h](file:///d:/ZJY/GitHub/ToTheMoon_PuzzleGame/gui/include/Tool.h#L40-L47)）：

| 方法 | 功能 | 动画属性 |
|------|------|----------|
| `createOverlay()` | 创建透明覆盖层 `QWidget`，挂载 `QGraphicsOpacityEffect` | 初始 `opacity = 0.0`，不拦截鼠标事件 |
| `createOpacityAnimation()` | 对 `QGraphicsOpacityEffect` 的 `opacity` 属性创建属性动画 | 透明度渐变动画 |
| `createScaleAnimation()` | 对 `QWidget` 的 `geometry` 属性创建属性动画 | 几何缩放动画 |

三个方法均接受起始值、终止值、持续时间（ms）和缓动曲线参数，返回 `std::unique_ptr<QPropertyAnimation>`，交由调用方按需组合为 `QSequentialAnimationGroup`（顺序动画组）或 `QParallelAnimationGroup`（并行动画组）。

**（2）图片缓存预加载**

翻转动画需要在短时间内频繁切换格子的正面/反面图片。如果每次切换都实时从磁盘读取并切割 `QPixmap`，会造成明显的 I/O 卡顿，导致动画掉帧。为此，`BoardView` 在游戏启动时通过 `preloadPiecePixmaps()` 将所有格子的图片预先切割并缓存到二维数组 `cellOnPiecePixmaps[6][6]` 和 `cellOffPiecePixmaps[6][6]` 中：

```cpp
void BoardView::preloadPiecePixmaps(){
    int onPieceWidth = cellOnPixmap.width() / 6;
    int onPieceHeight = cellOnPixmap.height() / 6;
    for (int row = 0; row < 6; row++) {
        for (int col = 0; col < 6; col++) {
            int xOn = col * onPieceWidth;
            int yOn = (5 - row) * onPieceHeight;
            cellOnPiecePixmaps[row][col] =
                cellOnPixmap.copy(xOn, yOn, onPieceWidth, onPieceHeight);
            // cellOffPiecePixmaps 同理
        }
    }
}
```

- 整张大图（如"To The Moon"主题图）被均匀切割为 6×6=36 块，每块对应一个格子位置的正面图案
- 动画期间 `updateSingleCell()` 直接从缓存数组取用，无需任何磁盘 I/O 或图片缩放操作
- 动画结束后 `updateBoardDisplay()` 同样复用缓存，保证全量刷新时也不会卡顿
- 缓存在每次 `startGame()` 时根据当前模式（随机/关卡）重新生成，确保不同主题图正确加载

**（3）翻转动画（Turn Animation）**

玩家每次操作（翻转行/列/对角线）后，受影响的格子会逐格播放卡片翻转效果。实现分为三个阶段：

**阶段一：变化检测**（`startTurnAnimation()`）

```cpp
void BoardView::startTurnAnimation(){
    // 保存操作前的拼图状态 → previousBoard
    // 操作后 → currentBoard
    for (int j=0; j<cols; j++){
        for (int i=0; i<rows; i++){
            if (previousBoard[i][j] != currentBoard[i][j]){
                cellsToAnimate.push_back({i, j});
            }
        }
    }
    animationTimer->start(200);  // 每 200ms 播放下一个格子的动画
}
```

在每次用户操作前，通过 `previousBoard = controller->getBoard()` 保存快照；操作后逐格比对，将状态发生变化的格子加入 `cellsToAnimate` 列表。动画期间所有操作按钮被禁用（通过 `setButtonsEnabled(false, false)`），防止玩家在动画播放时进行额外操作。

**阶段二：逐格动画**（`animateNextCell()` + `addAnimation()`）

`QTimer` 每 200ms 触发一次 `animateNextCell()`，依次处理 `cellsToAnimate` 中的每个格子：

```
animateNextCell() 的逐格处理流程：
┌─────────────────────────────────────────────────────┐
│  1. 取出当前格子坐标 (i, j)                           │
│  2. updateSingleCell(i, j) → 更新该格子的图片          │
│  3. addAnimation(btn)   → 播放在该格子上层的覆盖动画    │
│  4. currentAnimationIndex++                          │
│  5. 若全部播完 → 停止定时器，恢复棋盘显示，启用按钮      │
└─────────────────────────────────────────────────────┘
```

每个格子的 `addAnimation()` 创建一个**卡牌翻转效果**，由 4 个子动画顺序播放：

```
       ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
开始 → │  黑色覆盖层  │ → │   宽度缩小   │ → │   宽度放大   │ → │  黑色覆盖层  │ → 结束
       │  淡入 100ms  │    │  至 0 200ms  │    │ 至原宽200ms  │    │  淡出 100ms  │
       └──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
             中间点（此时更新为翻转后的图片）
```

实现细节：

- 在每个格子按钮上方覆盖一个黑色半透明 `QWidget`（`createOverlay`）
- 淡入（opacity 0.0 → 0.5, 100ms, InQuad）—— 卡片变暗
- 宽度缩小（geometry 宽度 → 0, 200ms, InOutQuad）—— 卡片"合上"
- 宽度放大（geometry 宽度 0 → 原宽, 200ms, InOutQuad）—— 卡片"翻开"（此时图片已更新）
- 淡出（opacity 0.5 → 0.0, 100ms, OutQuad）—— 覆盖层消失
- 动画组播放完毕后，通过 `deleteLater` 自动清理覆盖层

**阶段三：恢复状态**

所有格子动画播放完毕后，调用 `updateBoardDisplay()` 全量刷新、`checkGameState()` 检查胜负、`setButtonsEnabled(true, false)` 恢复按钮交互。

若用户在设置中关闭了动画特效，`startTurnAnimation()` 会直接走 else 分支：仅调用 `updateBoardDisplay()` + `checkGameState()`，跳过所有动画逻辑。

**（4）完成动画（Finish Animation）**

当玩家成功将全部碎片翻至正面时，触发胜利动画 `startFinishAnimation()`，由并行动画组同时执行两个独立效果：

**效果一：边框扩散动画**（`borderAnimation()`）

```
       ┌──────────────┐    ┌──────────────────────┐    ┌──────────────┐
开始 → │  白色边框    │ → │ 边框向外扩展 +       │ → │  边框渐隐    │ → 结束
       │  淡入 300ms  │    │ 边框加粗(10→60px)   │    │  淡出 300ms  │
       └──────────────┘    │ 缩放+增粗 700ms      │    └──────────────┘
                           └──────────────────────┘
```

- 在拼图板外围创建一个白色细线边框 `QWidget`，初始透明度为 0
- 淡入至 opacity=0.5 后，通过 `QParallelAnimationGroup` 同时执行：
  - 边框宽度动画（`borderWidth` 属性 10px → 60px, 700ms, OutQuad）
  - 几何缩放动画（边框从拼图大小向外扩展 250px, 1000ms, OutQuad）
- 最后边框淡出消失

**效果二：全屏闪光动画**（`shineAnimation()`）

```
       ┌──────────────┐    ┌──────────────┐
开始 → │  白色全屏    │ → │  白色全屏    │ → 结束
       │  淡入 200ms  │    │  淡出 200ms  │
       └──────────────┘    └──────────────┘
```

- 创建一个白色全屏覆盖层，透明度从 0 → 0.9（InQuad, 200ms）再 0.9 → 0.0（OutCubic, 200ms），形成一次快速的白色闪光

两个效果通过 `QParallelAnimationGroup` 并行播放，动画结束后自动清理所有临时 Widget。

## 六. 收获

### 1. 虚析构函数的重要性

在 `GameMode` 基类中，必须将析构函数声明为虚函数：

```cpp
class GameMode {
public:
    virtual ~GameMode() = default;
};
```

在 `BoardController` 中，使用基类指针管理派生类对象：

```cpp
void BoardController::setGameMode(GameMode* mode){
    if (gameMode) delete gameMode;  // 通过基类指针析构
    gameMode = mode;
}
```

如果 `~GameMode()` 不是虚析构函数，当通过基类指针 `delete` 派生类对象时，只会调用基类的析构函数而不会调用派生类的析构函数，导致**内存泄漏**（派生类中 `levelStore` 等成员不会被正确释放）。这是 C++ 中一个经典且极其重要的问题：**任何可能作为基类被继承的类，其析构函数必须声明为虚函数**。

### 2. 面向对象设计 —— 多态和原型模式

通过抽象基类 `GameMode` 定义统一接口，`RandomMode` 和 `LevelMode` 分别实现 `createBoard()` 等虚函数。在 `BoardController` 中通过基类指针调用，实现了运行时多态：

```cpp
// 基类指针，运行时指向 RandomMode 或 LevelMode
GameMode* gameMode;

// 多态调用 —— 无需知道具体是哪种模式
PuzzleBoard board = gameMode->createBoard();
```

同时，利用 `clone()` 虚函数实现了**原型模式**，可以方便地复制当前游戏模式而不需要知道其具体类型。

### 3. GF(2) 域高斯消元算法的理解

这是本次大作业最有收获的部分。将拼图翻转问题转化为 GF(2) 域上的线性方程组是一大思维突破：

- 理解到翻转操作的本质是**异或**运算，而 GF(2) 域上加法就是异或
- 掌握了 GF(2) 域上的高斯-约旦消元法 —— 与实数域消元类似，但使用异或代替加减法
- 自由变量枚举中利用**位掩码**（mask）的技巧非常高效：一个整数即可表示所有自由变量的赋值组合，配合剪枝策略大幅减少计算量

### 4. MVC 架构分离的价值

将 core 层（纯 C++ 游戏逻辑）与 gui 层（Qt6 界面）分离带来了以下好处：

- **可测试性**：core 层不依赖 Qt，可以用标准的 C++ 单元测试框架单独测试，无需启动 GUI
- **可移植性**：游戏核心逻辑可以复用到其他 GUI 框架（如控制台版本）或平台
- **职责清晰**：PuzzleBoard 只关心拼图状态和算法，BoardView 只关心渲染和动画，BoardController 作为桥梁协调两者

### 5. 单元测试的价值

编写单元测试时发现了一个潜在问题：`randomBoard()` 方法在打乱拼图后计算 `initialMinStep`，如果最优解步数太少（< 3 步），会递归重新打乱。如果单元测试没有覆盖这个边界条件，可能导致无限递归的风险没有被及时发现。单元测试帮助在早期定位并验证了这个逻辑的正确性。

此外，在对高斯消元算法进行优化时，单元测试能够快速回归验证：修改消元逻辑后，只需运行测试即可确认已知拼图的最优解仍然正确，而不需要手动逐一验证。

### 6. Qt 框架入门

本项目使用 Qt6 作为 GUI 框架，是第一次系统性地使用 Qt 进行完整的桌面应用开发。通过本项目，掌握了以下 Qt 核心知识：

- **信号与槽机制**：Qt 的核心通信方式，通过 `connect()` 将按钮点击、定时器超时等信号连接到对应的槽函数，实现了松耦合的事件驱动编程。例如倒计时 `QTimer::timeout` → `onCountdownTick()`、动画定时器触发 → `animateNextCell()`
- **QWidget 窗口与布局**：使用 `QStackedWidget` 实现主页、难度选择、拼图界面等多页面切换；通过 `.ui` 文件可视化设计界面布局，结合代码动态控制控件的可见性和启用状态
- **QPropertyAnimation 动画系统**：通过 `QPropertyAnimation` 对控件的 `opacity`、`geometry` 等属性实现渐变动画，结合 `QSequentialAnimationGroup` / `QParallelAnimationGroup` 组合复杂动画序列
- **QMediaPlayer 多媒体**：利用 `QMediaPlayer` + `QAudioOutput` 实现背景音乐和音效的播放、暂停、停止及音量控制
- **QPixmap 图像处理**：图片的加载、切割（`copy()`）、缩放（`scaled()`）以及通过 `QPalette` 设置窗口背景
- **CMake + Qt 构建**：使用 `qt_add_executable()` 和 `CMAKE_AUTOMOC/AUTOUIC/AUTORCC` 简化 Qt 项目的构建配置
